# 缺陷分级与置信度评分

本文档定义缺陷严重等级和置信度评分策略。

---

## 一、缺陷严重等级

### 缺陷类型分层

在判定 P0/P1/P2 之前，先将发现的问题归入以下四类之一。只有前两类进入 P0/P1 候选，后两类有严格的等级上限。

| 类型 | 定义 | 允许的最高等级 | 判定要点 |
|------|------|----------------|----------|
| **核心功能缺陷** | 直接导致程序行为错误、崩溃、数据损坏或安全漏洞的确定性 bug | P0 | 能描述出具体的、必然或高概率触发的执行路径 |
| **条件性功能缺陷** | 在特定输入、时序或边界条件下才触发的功能性 bug | P1 | 能描述出具体的触发条件（特定输入值、并发时序、数据量级等） |
| **防御性/健壮性不足** | 缺少防御性检查，但当前调用路径未触发问题，属于潜在风险而非已知 bug | P2 | 当前所有调用方均满足被调函数的前置条件，缺少检查仅增加未来风险 |
| **代码质量/风格问题** | 不影响功能正确性，仅影响可维护性或可读性 | 不报告 | 函数过长、魔法数字、命名不规范等 |

#### 分类决策规则

1. **必须有明确的触发路径才能归为核心功能缺陷**：如果无法在代码中找到一条具体的执行路径使问题必然或高概率发生，不得归为 P0。"理论上可能触发"不等于"核心功能缺陷"。
2. **条件性缺陷必须能描述出具体触发条件**：仅凭"如果传入异常数据可能出错"不足以判定为条件性功能缺陷（P1），必须指出具体的输入值、时序窗口或数据规模。
3. **防御性不足与功能缺陷的分界线**：如果当前所有调用方都满足被调函数的前置条件（如不传 nil、不传空字符串），则缺少检查属于防御性不足（最高 P2），不是功能缺陷（P0/P1）。判断时需读取直接调用方的代码确认。
4. **测试代码中的问题单独评估**：测试文件中的问题只有在影响测试可靠性（如断言恒通过、mock 遮盖真实行为导致测试无法检测回归）时才报告为 P1，其余降为 P2 或不报告。

---

### P0 — 阻断（必须修复后才能上线）

会导致数据损坏、服务崩溃、安全漏洞或严重业务错误的问题。

典型特征：
- 生产环境必现或高概率触发
- 影响范围广（影响多用户或核心流程）
- 无法通过业务层规避

示例：
- 并发写共享 map 导致 panic
- SQL 注入漏洞
- nil 指针解引用在必经路径上
- 幂等缺失导致重复扣款
- 越权访问/修改他人资源（IDOR）：调用者可通过接口操作不归属自己的数据

不属于 P0 的情况：
- 仅在理论上可能触发但无法构造具体触发路径的问题
- 防御性编程缺失但当前调用链未暴露风险（属于"防御性/健壮性不足"，最高 P2）
- 测试代码中的问题（除非导致测试恒通过，遮盖真实缺陷）

### P1 — 严重（应在本次迭代修复）

特定条件下会触发的错误，影响功能正确性，但不会立即导致服务崩溃。

典型特征：
- 需要特定输入或时序才能触发
- 影响单个功能或部分用户
- 可能在压力/边界场景下暴露

示例：
- 错误被吞掉导致问题无法排查
- 数据库连接未关闭（高并发下连接耗尽）
- 整数溢出在特定输入时触发
- goroutine 泄漏（长期运行下累积）

不属于 P1 的情况：
- 纯代码风格或可维护性问题
- 错误信息不够详细（除非直接导致关键生产问题无法排查）
- 缺少防御性检查但所有调用方已满足前置条件（属于"防御性/健壮性不足"，P2）

### P2 — 一般（建议修复）

不影响核心功能正确性，但存在潜在风险或影响可维护性。

典型特征：
- 触发概率低或影响范围小
- 可通过其他机制（监控、重试）缓解

示例：
- 错误信息缺少上下文，排查困难
- 函数过长但逻辑尚清晰
- 硬编码配置值（非密钥）

### 定级规则

- 一个问题符合多个级别特征时，**向上取整**（按最高级别定）
- **安全类问题最低 P1**，涉及认证/授权/数据泄露直接 P0
- 纯代码风格和命名问题不报告（原 P3 级别已取消）

### 定级自检清单

为每个缺陷标定严重等级前，逐项确认：

1. **触发路径**：能否描述出具体的、可复现的触发路径？（否 → 降级或降低置信度）
2. **影响性质**：问题是否直接导致功能错误，还是仅增加潜在风险？（仅增加风险 → 最高 P2）
3. **调用方前置条件**：当前所有调用方是否已满足被调函数的前置条件？（是 → 防御性不足，最高 P2）
4. **代码类型**：是否属于测试代码中的问题？（是 → 按测试代码标准单独评估）

## 二、置信度评分策略

置信度用 1-10 分表示对缺陷判定的确信程度。

| 分段 | 含义 | 判定标准 |
|------|------|---------|
| 9-10 | 确定 | 仅从代码即可确认的确定性缺陷，无歧义 |
| 7-8 | 高 | 极大概率是缺陷，但依赖运行时状态或调用上下文 |
| 5-6 | 中 | 合理怀疑，需要了解更广泛的系统上下文才能确认 |
| 3-4 | 低 | 可疑模式，但可能是有意为之，需人工确认 |
| 1-2 | 推测 | 基于惯例的猜测，高误报风险 |

使用规则：
- 置信度 < 5 的缺陷**不报告**，除非严重等级为 P0
- 置信度 5-6 的缺陷，在 rationale 中加注"需结合业务场景确认"
- 同一严重等级的缺陷，按置信度降序排列

### 外部契约依赖降信

当缺陷判定依赖对外部系统行为的假设（如上游服务已校验、其他微服务已加密、中间件保证 exactly-once、基础设施提供全局超时等），置信度应主动下调。完全依赖外部假设的问题置信度不超过 4，部分依赖的置信度 -3（最低 1）。此类问题不应直接判为 P0。
