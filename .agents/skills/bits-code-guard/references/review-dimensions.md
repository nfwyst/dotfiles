# 评审维度

本文档定义通用检测模式的 6 个评审维度及检测清单。

---

## LOGIC — 逻辑错误

代码逻辑不正确，会导致错误结果或非预期行为。

检测清单：
- 条件分支是否覆盖所有情况（switch/if-else 的 default 分支）
- 循环的起始、终止条件和步进是否正确，避免死循环或少/多迭代
- 函数是否在所有代码路径上都有返回值
- 不可达代码（return 后的语句、永远为 false 的条件）
- 布尔表达式是否正确（&&/|| 混用、取反遗漏）
- 比较运算符方向是否正确（> vs <、>= vs >）
- 字段名/变量名是否笔误（逐个比对左右相近但不完全相同的名称，如 `userID` 误写为 `orderID`）
- 函数调用传参是否正确（参数顺序、类型是否混淆）
- 返回值变量是否误用为另一个相关变量
- 与其他位置相似代码是否存在不一致（copy-paste 修改不完全）
- 时间与日期逻辑是否可靠（时区假设是否成立、时间戳单位秒/毫秒是否混用、跨时区场景下的计算是否正确）
- 缓存与数据源之间的一致性（写入后缓存是否同步更新/失效、缓存 key 的构造是否稳定可预测、读写路径是否对称）
- 接口或抽象方法的实现是否满足约定（返回值类型、行为语义是否与声明一致）
- 相等性判断是否符合预期（值比较 vs 引用比较、字符串比较是否需要忽略大小写）
- 注释与实现一致性：代码中的注释/文档字符串声明了某种前置条件或后置保证（如"确保无 X 引用"、"只允许 Y"），实现是否真的做了
- 参数被接收但未使用：函数/方法签名接收了参数但函数体未读取该参数（排除 interface 实现占位）
- 互斥字段未约束：同一对象上存在语义互斥的多个 bool/enum 字段（IsA/IsB/IsC），写入路径是否校验同一时刻只能有一个为真
- 错误/早退分支处理完后是否及时通过 return/break/continue 中断控制流，避免继续执行后续的成功路径逻辑（如错误已记录但未 return，导致成功日志/计数被同时写入）
- 字符串经过拼接/裁剪/解析后，结果的语义是否仍与原始输入一致（如截断是否破坏了多字节字符、trim 是否误删了有意义的分隔符）
- 返回集合是否有稳定的顺序（避免直接返回 map 迭代结果等非确定顺序）
- 错误消息描述的失败原因是否与实际校验的条件一致（如代码实际校验的是字段 A 为空但消息提示 "missing B"），错误消息误导会显著增加排查成本

## SECURITY — 安全漏洞

可能被利用的安全缺陷。

检测清单：
- SQL 拼接（未使用参数化查询）
- 命令注入（`exec.Command` / `os/exec` 直接拼接用户输入）
- 路径遍历（未过滤 `../` 的文件路径操作）
- 硬编码密钥/token/密码
- 敏感信息写入日志或对外响应体（内部错误详情、DB 报错、堆栈信息不应出现在 HTTP/RPC 响应中，应记录日志并返回统一业务文案）
- 凭证存储反模式：密码、token、密钥写入持久化存储前是否经过哈希/加密（寻找 hash/bcrypt/argon/crypt/encrypt 调用，缺失即可疑）
- 不安全的反序列化
- XSS（未转义用户输入直接输出到 HTML）
- SSRF（未校验的 URL 请求）
- 对外接口是否遗漏身份验证（未登录即可访问受保护资源）
- 权限控制：（IDOR）：每个需要授权的操作，执行前是否同时验证了"资源存在"和"资源归属当前调用者"；仅凭资源 ID 能定位记录不等于有权操作该记录
- 跨资源操作中，关联资源是否同时验证了存在性与归属当前调用者（仅验证关联 ID 存在不够，还需确认该资源属于当前用户）
- 文件/二进制上传接口是否校验实际内容类型（Content-Type 与文件签名/magic number），仅凭扩展名或声明类型不足以防止伪装

## CONCURRENCY — 并发问题

竞争条件、死锁、goroutine/线程泄漏等。

检测清单：
- 共享变量是否有互斥保护（mutex、atomic、channel）
- map/slice 在多 goroutine 中是否并发读写
- 单例初始化是否线程安全
- 锁的粒度是否合理（过大影响性能，过小有竞争风险）
- goroutine/线程 是否有退出条件，避免泄漏
- channel 是否存在死锁风险（发送方/接收方不匹配）
- 是否存在检查-使用窗口问题（先判断条件再执行操作，但两步之间状态可能已被其他线程/协程改变）

## ROBUSTNESS — 健壮性

空指针、越界、未处理错误、资源未释放等防御性编程问题。

检测清单：
- 指针/引用解引用前是否检查 nil/null
- 数组/切片访问是否有越界风险
- map 访问前是否检查 key 存在
- 函数返回值（特别是 error）是否全部被检查
- 整数运算是否存在溢出风险
- 业务边界值处理：字段校验是否与其业务语义一致，对该类型数据的常见攻击面做了防御（如字符串的空值/超长、数值的非法范围、零值/负数等边界）
- 枚举或受限集合字段是否在进入业务逻辑前做白名单拦截，避免非法值穿透到下游存储或判断
- 分页或批量接口的 PageSize/Limit 类参数是否同时设置了下限和上限，避免被恶意超大值拖垮存储/内存
- 业务层会直接解引用的嵌套指针，校验函数是否提前拦截了 nil 值，避免未校验的嵌套字段在下游引发 panic
- 文件/连接/锁 打开后是否有对应的 close/release
- 数据库连接/事务是否正确关闭/提交/回滚
- 错误是否被吞掉（`_ = err` 或空 catch）
- 错误信息是否包含足够上下文
- panic/recover 使用是否合理
- HTTP/RPC 错误码是否正确映射：不同错误类型（资源不存在、调用者无权限、业务约束冲突、内部异常）应映射到不同状态码，不可因错误处理路径合并而统一返回内部错误
- 异步调用或后台协程的执行结果是否被正确处理（避免启动后不关心成败的"发射后不管"模式）
- 类型转换和断言是否安全（如 Go 的 type assertion 是否使用 comma-ok 模式、强制转换是否可能丢失精度）

## PERFORMANCE — 性能问题

明显的性能缺陷，不包含微优化建议。

检测清单：
- N+1 查询（循环内执行数据库查询）
- 大对象在循环中反复分配/复制
- 无效循环（可提前 break 或用更高效的数据结构）
- 未使用缓冲的 I/O 操作
- 不必要的序列化/反序列化

## QUALITY — 代码质量

可维护性问题，不影响功能正确性。此维度优先级最低，只在问题非常明显时报告。

检测清单：
- 函数过长（>80 行）且逻辑复杂
- 重复代码块（3 次以上相同逻辑）
- 魔法数字/字符串未定义为常量

---

## 误报规避

以下情况不应报告为缺陷：

- **框架/库约定**：框架内部看似异常的代码模式，先确认是否为框架约定
- **测试代码简化**：`_ = err` 在测试文件中通常可接受
- **注释代码**：除非明确是遗留死代码
- **风格差异**：只有唯一正确写法的情况才报告
- **空指针猜测**：报告空指针问题必须有明确的空值来源代码证据（如 API 输入指针、RPC 返回指针、显式赋 nil），严禁仅凭"可能为空"进行猜测

对于不确定的问题，在 rationale 中注明"需结合业务场景确认"，置信度相应降低。
